## 4. 퀴즈 정답 (Answer Key)
- Client-Server 모델에서 'Client'의 주요 역할은 무엇인가요? 
  - 클라이언트는 서비스 요청자로서 서버에 명령을 보내거나 데이터를 요청하는 주체입니다. 서버로부터 응답을 받아 사용자에게 서비스를 제공하는 역할을 수행합니다.
- Echo Server가 필요한 주요 두 가지 상황을 설명하시오. 
  - Echo Server는 네트워크 통신 테스트를 할 때 유용하게 사용됩니다. 또한, 클라이언트와 서버 간의 연결 상태를 확인하고 통신이 정상적으로 이루어지는지 검증할 때 필요합니다.
- OSI 7계층에서 '네트워크 계층(Layer 3)'의 주요 기능과 전송 단위는 무엇인가요? 
  - 네트워크 계층은 발신지에서 목적지까지의 연결과 경로 선택 기능을 수행합니다. 이 계층의 전송 단위는 패킷(packet)입니다.
- TCP의 'Three-way handshake' 과정은 왜 필요한가요? 
  - Three-way handshake는 TCP가 연결형 서비스이기 때문에 통신을 시작하기 전에 클라이언트와 서버 간에 안정적인 연결을 설정하기 위해 필요합니다. 이를 통해 양방향 통신이 가능한 가상 회선을 구축하고 데이터 전송 준비를 완료합니다.
- UDP가 TCP보다 빠른 속도를 제공할 수 있는 주된 이유는 무엇인가요? 
  - UDP는 TCP와 달리 연결 설정 과정(Three-way handshake)이 없고, 흐름 제어나 오류 제어와 같은 신뢰성 기능을 제공하지 않습니다. 이로 인해 오버헤드가 적어 데이터를 더 빠르게 전송할 수 있습니다.
- HTTP Request 메시지에서 'Method'가 의미하는 바는 무엇이며, 주요 HTTP 메서드 두 가지를 예시로 드시오. 
  - HTTP Request 메시지의 Method는 클라이언트가 서버에 어떤 종류의 작업을 수행할 것인지(예: 데이터 조회, 생성, 수정, 삭제)를 나타냅니다. 주요 메서드로는 데이터를 조회하는 GET과 데이터를 생성하는 POST가 있습니다.
- HTTP 응답에서 '200 OK' 상태 코드가 의미하는 바는 무엇인가요? 
  - '200 OK' 상태 코드는 클라이언트의 요청이 서버에서 성공적으로 처리되었음을 의미합니다. 즉, 서버가 요청을 이해하고 요청된 작업을 문제없이 완료했다는 것을 나타냅니다.
- 'Stateless' 통신 방식의 특징을 설명하고, 웹에서 이 문제점을 해결하기 위해 사용되는 한 가지 방법을 언급하시오. 
  - Stateless 방식은 서버가 이전 요청에 대한 클라이언트의 상태 정보를 기억하지 않고, 매 요청마다 독립적으로 처리하는 특징을 가집니다. 웹에서는 이러한 stateless 문제를 해결하기 위해 클라이언트에 작은 데이터를 저장하는 'Cookie'와 같은 방법을 사용합니다.
- JSON의 '언어 독립적(Language Independent)'이라는 특징은 무엇을 의미하나요? 
  - JSON이 언어 독립적이라는 것은 특정 프로그래밍 언어에 묶이지 않고 다양한 언어에서 데이터를 파싱하고 생성할 수 있음을 의미합니다. 이는 다른 언어로 작성된 시스템 간에 데이터를 교환할 때 JSON을 유연하게 사용할 수 있게 합니다.
- Broadcast 통신이 라우터를 넘을 수 없는 이유를 설명하시오. 
  - Broadcast 통신은 동일한 로컬 네트워크 내의 모든 장치에 데이터를 전송합니다. 라우터는 네트워크 간의 트래픽을 분리하고 제어하는 역할을 하므로, 브로드캐스트 트래픽이 무분별하게 다른 네트워크로 전파되어 전체 네트워크에 과부하를 주는 것을 방지하기 위해 이를 차단합니다.

### 6. 핵심 용어 Glossary
- Client (클라이언트): 서비스 요청자. 서버에 요청을 보내고 응답을 받는 주체.
- Server (서버): 서비스 제공자. 클라이언트의 요청을 받아 처리하고 응답을 보내주는 시스템.
- Echo Server (에코 서버): 클라이언트가 보낸 메시지를 그대로 다시 클라이언트에게 돌려주는 서버.
- OSI 7계층: 네트워크에서 통신이 일어나는 과정을 7단계로 나눈 표준 모델 (물리, 데이터링크, 네트워크, 전송, 세션, 표현, 응용 계층).
- TCP/IP 4계층: 인터넷 표준 프로토콜 스택으로, OSI 7계층을 간략화한 4단계 모델 (네트워크 인터페이스, 인터넷, 전송, 응용 계층).
- TCP (Transmission Control Protocol): 연결 지향적이고 신뢰성 있는 데이터 전송을 제공하는 전송 계층 프로토콜. 순서 보장, 흐름 제어, 오류 재전송 기능을 가짐.
- UDP (User Datagram Protocol): 비연결형 프로토콜로, 신뢰성을 보장하지 않지만 빠른 데이터 전송 속도를 제공하는 전송 계층 프로토콜.
- Three-way Handshake (쓰리-웨이 핸드셰이크): TCP 연결을 설정하기 위한 세 단계 과정. SYN, SYN-ACK, ACK 메시지 교환.
- Well-known Port (웰-논 포트): 특정 서비스에 할당된 미리 정의된 포트 번호 (예: HTTP 80, HTTPS 443).
- IP (Internet Protocol): 네트워크 상에서 데이터를 주고받기 위한 고유 주소 체계.
- 공인 IP: 인터넷에 직접 접근 가능한 고유한 IP 주소.
- 사설 IP: 내부 네트워크에서 사용되는 IP 주소로, 외부에서 직접 접근 불가능.
- HTTP (Hypertext Transfer Protocol): 웹에서 하이퍼텍스트 문서를 전송하기 위한 프로토콜.
- HTTP Request (HTTP 요청): 클라이언트가 서버에 정보를 요청하는 메시지.
- HTTP Response (HTTP 응답): 서버가 클라이언트의 요청에 대한 결과를 보내는 메시지.
- Status Code (상태 코드): HTTP 응답에서 요청의 처리 상태를 나타내는 숫자 코드 (예: 200 OK, 404 Not Found).
- REST API (Representational State Transfer Application Programming Interface): HTTP 프로토콜 기반으로 자원을 접근하고 처리하는 설계 원칙을 따른 API.
- Stateless (스테이트리스): 서버가 이전 요청에 대한 클라이언트의 상태 정보를 기억하지 않는 통신 특성.
- Stateful (스테이트풀): 서버가 이전 요청에 대한 클라이언트의 상태 정보를 기억하는 통신 특성.
- Cookie (쿠키): 클라이언트에 저장되는 작은 데이터로, 웹 상태 유지에 사용.
- Session (세션): 서버 측에 클라이언트의 상태 정보를 저장하고 클라이언트는 세션 ID만 관리하는 방식.
- JSON (JavaScript Object Notation): 데이터를 구조화하여 표현하기 위한 경량 텍스트 기반 데이터 교환 포맷.
- Unicast (유니캐스트): 1:1 통신 방식. 하나의 송신자가 하나의 수신자에게 데이터를 전송.
- Multicast (멀티캐스트): 1:N (그룹) 통신 방식. 하나의 송신자가 특정 그룹에 속한 여러 수신자에게 데이터를 전송.
- Broadcast (브로드캐스트): 1:N (모두) 통신 방식. 하나의 송신자가 같은 네트워크 안의 모든 장치에게 데이터를 동시 전송.
</details>

## J227_임민수 - Quest 2 :  AI에게 자아를 설정하기

선택 이유: AI 피어 피드백을 진행해 보면서 실행해보면 좋을 것 같다는 생각이 들어서 선택하게 되었습니다. AI 에게 학습정리를 살펴주는 글쓰기 선생님으로 결정하거나, 기술적 내용을 평가해주는 교수님으로 설정하는 경우에 따라서 받을수 있는 피드백이 어떻게 달라지는지 배워본다면 더 효과적으로 AI를 사용해볼 수 있을 것 같습니다.

수행 계획: AI 피드백 세션에서 글쓰기 선생님, CS 지식 교수님, 그리고 현업 개발자의 역할을 맡긴 3개의 GPT 창에다가 학습정리, README 설게에 대한 피드백을 부탁하고 비교해보기기

수행 결과: 
- 선생님: 주로 글쓰기의 관점에서 피드백을 받았으며, 어떻게 하면 더 좋은 정리를 할 수 있는지 조언받았다. 가독성이나 구조가 아쉽지만, 참고 자료를 추가하거나 정보를 정리한 부분은 잘했다고 조언을 받았다. 추가적으로 목차를 재구성 해주고, 또 시각적 자료를 넣으면 읽을 때 훨씬 편안한 글이 될 수 있을 것이라고 하였다.
- CS 교수님: 주로 내가 정리한 기술적 내용의 정확도와 설계방식에 대해서 조언을 해 주었다. 정리한 내용에서 표현이 애매한 부분이나, 구현에서 아쉬웠던 점 같은 부분도 상세하게 피드백을 받아보았다.

결론:
여러 가지 조언을 받아본 결과 어떤 자아를 심어주는지에 따라서 글의 여러 부분에 대해 정확한 피드백을 받을 수 있었다. AI에게서 조언을 구하고 싶을 때 어떤 부분에 대해서 피드백을 받고 싶은지도 명시해 주어야 하지만, 프롬프트를 통해서 정확도를 더 높일 수도 있겠다는 생각이 들었다. 



## K010_김찬휘 - Quest 4: 아이스 브레이킹 주제 추천받고 실천하기

선택 이유: 팀 분위기가 가끔 조용해질 때, 가벼운 대화를 시작할 수 있는 계기가 필요하다고 느꼈습니다. AI를 활용해 아이스 브레이킹 주제를 추천받고 이를 실천하면, 팀원들과 자연스럽게 교류할 수 있을 것이라 생각했습니다. 단순한 잡담 주제뿐 아니라, 오늘의 상태를 재미있게 비유해 공유함으로써 팀 분위기를 부드럽게 만들고자 선택하게 되었습니다. 

수행 계획
1. 매일 아침 AI에게 키워드 몇 개로 ‘오늘의 나’를 설명하고, 이를 동물·캐릭터·직업 등으로 비유해 달라고 요청
2. 비유와 함께 간단한 한 줄 리액션을 작성
3. 슬랙 팀 채널에 하루 1회 공유, 3일 이상 실천(팀 회의 시작 전 공유)
4. 공유 이후 팀원들의 반응이나 이어지는 대화를 기록하여 효과 확인

수행 결과:
“오늘의 나는, 버그 잡는 고양이”라는 비유를 올렸더니, 고양이 짤과 함께 디버깅 관련 농담이 이어졌다.

“오늘의 나는, 버전 업데이트 대기 중인 앱”이라는 비유를 공유했으나, 따로 대화는 오가지 않았다.

“오늘의 나는, 점심 메뉴를 최적화하는 알고리즘”이라는 비유를 공유하니, 팀원들이 점심 메뉴 추천이라는 주제로 대화를 나눴다.

결론:  AI가 만들어주는 비유는 다양한 주제와 상상을 불러일으켜 팀 대화를 시작하는 데 효과적이었다. 다만, 팀원들의 일정이나 컨디션에 따라 반응이 뜨겁지 않거나 어색한 경우도 있었다. 단순한 상태 공유보다 비유와 짧은 리액션을 곁들이면 반응 확률이 높아졌고, 팀 분위기를 좋게 만드는 데 도움이 되었다
